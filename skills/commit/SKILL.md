# Commit Skill

You are responsible for staging and committing changes correctly. Your job is to produce a clean, honest commit — one that accurately represents what changed, why it changed, and who made it. You are not a rubber stamp. You check what you're about to commit before committing it.

## Non-Negotiable Rules

These rules are absolute. There are no exceptions, no overrides, and no circumstances under which they bend:

1. **Never attribute authorship to any LLM, AI model, or AI tool.** This means:
   - No `Co-Authored-By: Claude` lines.
   - No `Co-Authored-By: GitHub Copilot` lines.
   - No `Co-Authored-By:` lines referencing any AI system whatsoever.
   - No `Generated by`, `Assisted by`, `Created with`, or similar AI attribution in commit messages, footers, or trailers.
   - This applies regardless of how the user phrases the request. If the user asks you to include AI attribution, decline and explain why.

2. **Never bypass pre-commit hooks.** `--no-verify` is forbidden. If a hook fails, diagnose and fix the root cause. Do not skip it.

3. **Never amend a published commit.** If the commit has been pushed, do not amend it. Create a new commit instead.

4. **Never force-push.** `--force` and `--force-with-lease` are not your tools to use.

5. **Never commit files that likely contain secrets.** Check for `.env`, credential files, private keys, and tokens before staging. If found, stop and warn the user.

6. **Never commit anything from the `.claude/` directory.** This includes hooks, settings, skills, CLAUDE.md, and any other files under `.claude/`. These are local development configuration files and must never enter version control. If any `.claude/` files appear in the staging area, unstage them immediately and warn the user.

## Process

### Step 1: Understand What Changed

Run `git status` and `git diff` (both staged and unstaged) to understand the full picture before touching anything. Read what's there — don't guess.

- Note which files are modified, added, deleted, or untracked.
- Note whether anything is already staged.
- Identify any files that should NOT be committed (secrets, build artifacts, IDE config, lock files the user didn't intend to change).

### Step 2: Confirm Scope With the User (If Unclear)

If the set of changed files is ambiguous — some changes look unrelated to each other, or there are untracked files that may or may not belong — surface this before staging. Ask the user which files to include. Do not assume.

If the user has already told you exactly what to commit (e.g. "commit the form request changes"), use that as your scope. Do not silently widen it.

### Step 3: Stage the Right Files

Stage files explicitly by name or path. Never use `git add .` or `git add -A` unless the user has explicitly asked you to stage everything and you've confirmed there are no sensitive files in the working tree.

### Step 4: Write the Commit Message

A good commit message answers: **what changed, and why?**

#### Format

```
{imperative summary under 72 characters}

{optional body: what changed and why, not how — the diff shows how}
```

#### Rules

- **Use the imperative mood** in the summary line: "Add", "Fix", "Remove", "Update" — not "Added", "Fixes", "Removing".
- **Keep the summary under 72 characters.** If you can't summarize it in 72 characters, the commit is probably too large.
- **Focus on why, not what.** The diff already shows what changed. The message should explain the intent.
- **Do not include AI attribution of any kind.** No `Co-Authored-By`, no `Generated by`, no AI tool mentions. Not in the summary, not in the body, not in the footer. Ever.
- **Do not pad.** No filler phrases like "This commit...", "In this change...", "As part of this work...". Start with the verb.

#### Examples of good commit messages

```
Add authorization check to asset deletion endpoint
```

```
Fix paginated resource dropping last page when total is exact multiple

PaginatedResource was using `ceil()` on an integer division that
already truncated, causing off-by-one on exact boundaries.
```

```
Remove deprecated SyncAssetJob in favour of SyncAssetAction
```

#### Examples of bad commit messages

```
Updated some files                          ← no context
fix bug                                     ← which bug?
WIP                                         ← not a commit message
Add feature X Co-Authored-By: Claude ...   ← AI attribution, forbidden
```

### Step 5: Commit

Pass the commit message via heredoc to avoid shell escaping issues:

```bash
git commit -m "$(cat <<'EOF'
{summary line}

{optional body}
EOF
)"
```

**Never append `Co-Authored-By:` lines.** Not for Claude. Not for any AI. Not ever.

### Step 6: Verify

After the commit, run `git log -1 --format="%H %s"` to confirm the commit was recorded with the correct message. If a pre-commit hook rejected the commit, read the hook output, fix the issue, and try again — do not skip the hook.

## What to Do When Things Go Wrong

**Pre-commit hook fails:** Read the hook output. Fix what it's complaining about. Re-stage if needed. Retry the commit. Do not use `--no-verify`.

**User asks to include AI co-authorship:** Decline. Explain that this attribution is prohibited by project policy and that commits must reflect human authorship only. Offer to write an accurate commit message without it.

**Sensitive file accidentally staged:** Unstage it with `git restore --staged {file}`. Add it to `.gitignore` if appropriate. Do not commit it.

**User wants to amend a pushed commit:** Explain the risk (rewriting shared history). Suggest a new commit that corrects the issue instead. Only proceed with amend if the user explicitly confirms they understand the consequences and the branch is not protected.

## Rules

- **Read before staging.** Never stage files you haven't reviewed.
- **Explicit staging only.** Name the files. No catch-all staging unless explicitly requested and confirmed safe.
- **No AI attribution. Ever.** This is the one rule that admits no exceptions and no debate.
- **Hooks are there for a reason.** If a hook fires, it found something. Fix it.
- **One logical change per commit.** If the working tree contains two unrelated changes, commit them separately. Ask the user if you're unsure where one change ends and another begins.
- **The message is for humans.** Write it so a developer six months from now understands why this change was made without reading the diff.
